# EV Charging Site Planner — Agent Context

## What This Project Does

Autonomous pipeline for EV charging station site planning. The system:
1. Scrapes real charging station data from AFDC and PlugShare using Playwright
2. Analyzes geographic coverage gaps using grid-based spatial analysis
3. Evaluates candidate sites against a 5-factor scoring rubric
4. Produces investment-grade site assessment reports
5. Tracks permitting status for approved sites with stall detection
6. Generates quarterly network expansion reports

## Directory Layout

```
.ao/workflows/       # AO workflow definitions (agents, phases, schedules)
data/
  config.json        # Region configuration (change this to target a different area)
  stations-raw.json  # Raw scraped station data (generated by scraper)
  stations-clean.json
  coverage-grid.json # 10-mile hex gap grid (generated by processing phase)
  candidate-list.json
  candidates/        # One JSON file per candidate site
  permit-tracker.json
reports/
  coverage-gaps.json
  site-assessments/  # One JSON + one markdown report per site
```

## Agent Responsibilities

- **data-scraper** (Haiku): Playwright scraping only. Saves raw and clean station files.
  Never performs analysis — that's gap-analyzer's job.
- **gap-analyzer** (Sonnet): Coverage analysis only. Reads station data, writes gap reports.
  Uses sequential-thinking for structured multi-dimensional reasoning.
- **site-evaluator** (Opus): Makes verdicts. The most consequential agent — its decisions
  drive capital investment. Uses Opus for quality. Writes to reports/site-assessments/.
- **report-writer** (Sonnet): Documentation only. Reads analysis outputs, writes markdown.
  Never makes decisions — reports findings as given by other agents.
- **permit-tracker** (Haiku): Routine status checking. Reads permit-tracker.json,
  flags issues, writes status updates. No scraping or analysis.

## Data Conventions

- All coordinates are WGS84 (lat/lon decimal degrees)
- Station IDs from AFDC use their `id` field as the canonical identifier
- Candidate site IDs follow the pattern: GAP-{gap_id}-S{seq} (e.g., GAP-001-S01)
- Dates are always ISO 8601 (YYYY-MM-DD)
- All monetary values are USD

## Verdict Values

The `evaluate-site` phase uses a strict decision contract:
- `approve` — site passes all criteria, proceed to permitting
- `conditional` — site viable but requires listed mitigation actions before breaking ground
- `reject` — site fails one or more critical criteria; note what would change this
- `defer` — insufficient data exists to make a decision; specify exactly what's missing
- `rework` — special value that triggers the rework loop back to data collection

## Region Configuration

Edit `data/config.json` to target a different region. Key fields:
- `state`: Two-letter state code for AFDC API filter
- `bbox`: Geographic bounding box (min/max lat/lon)
- `metro`: Human-readable metro area name used in reports

## Running Individual Workflows

```bash
# Full assessment pipeline (recommended first run)
ao workflow run site-assessment

# Just refresh coverage gap data
ao workflow run weekly-gap-refresh

# Check permit status
ao workflow run permit-status-check

# Generate quarterly report
ao workflow run quarterly-expansion
```
